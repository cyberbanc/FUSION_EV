# M9 FUSION EV 1.2.0 — PancakeSwap Prediction paper bot

Полная версия paper-бота для PancakeSwap Prediction. Решение фиксируется в окне T−40, ставка создаётся каждый раунд, история хранится в PostgreSQL без автоматического удаления.

## Почему создана версия 1.2.0

Анализ 44 закрытых сделок показал две основные проблемы версии 1.1.0:

1. бот слишком часто менял более вероятную сторону на менее вероятную ради умеренного расчётного EV;
2. одна общая payout-коррекция UP/DOWN завышала будущий коэффициент, особенно когда raw-коэффициент был выше 3.00.

Версия 1.2.0 меняет именно эти части, сохраняя общую архитектуру и всю существующую историю.

## Новая последовательность выбора стороны

1. Рассчитываются вероятности Price/Chainlink, Binance и Crowd.
2. M9 и паттерны продолжают рассчитываться для диагностики, но по умолчанию имеют вес `0`.
3. Raw payout корректируется по стороне и диапазону коэффициента.
4. Если лучший corrected EV не ниже `30%`, EV-кандидат может использоваться.
5. Если EV-кандидат противоречит основной вероятности, он допускается только при:
   - EV не ниже `35%`;
   - agreement выбранной стороны не ниже `30%`.
6. При слабом или отрицательном EV:
   - если Crowd и Binance независимо совпадают, используется их общая сторона;
   - иначе используется сторона основной вероятности.
7. Ставка создаётся каждый раунд.

## Payout-калибровка по диапазонам

Калибровка ведётся отдельно для UP и DOWN в диапазонах raw-коэффициента:

```text
< 1.50
1.50-2.00
2.00-2.50
2.50-3.00
>= 3.00
```

Каждый диапазон имеет собственные:

- начальную безопасную коррекцию;
- максимальную коррекцию;
- lower-quantile по фактическим финальным пулам;
- количество наблюдений и readiness.

Для диапазона `>=3.00` по умолчанию действуют жёсткие пределы:

```text
UP   <= 0.60
DOWN <= 0.70
```

Эти значения можно изменить позже через Railway Variables, но на первом независимом тесте менять их не следует.

## Веса моделей 1.2.0

```text
Price / Chainlink = 0.40
Binance           = 0.25
Crowd             = 0.35
M9                 = 0
Patterns           = 0
```

M9 и паттерны остаются в `components_json`, поэтому их качество продолжает накапливаться и может анализироваться, но они не влияют на сигнал.

## Ставки

В Railway по умолчанию:

```text
VARIABLE_STAKE_ENABLED=false
```

Поэтому бот ставит только `$5`, независимо от банка, серии результатов и расчётного EV.

Код поддерживает будущий режим:

- `$10` — после минимум 200 сделок, готового payout-bucket, EV от 30% и agreement от 60%;
- `$15` — после минимум 300 сделок, EV от 50% и agreement от 70%.

Но включать его до отдельной проверки версии 1.2.0 не нужно.

## История и версии стратегии

PostgreSQL не очищается. При первом запуске автоматически добавляются:

- `strategy_version`;
- `payout_bucket_up`;
- `payout_bucket_down`.

Новые сделки получают:

```text
strategy_version = 1.2.0
```

Это позволяет анализировать новую версию отдельно:

```text
/strategy/performance
```

Старые сделки остаются в общей истории и не влияют на сохранность банка.

## Endpoint

```text
/health
/signal
/status?limit=30
/status?history=all&limit=100000
/history/export.csv
/payout/calibration
/model/performance
/strategy/performance
```

`/payout/calibration` теперь показывает таблицы диапазонов отдельно для UP и DOWN.

## Новые поля решения

- `strategy_version` — версия логики, создавшая решение;
- `payout_bucket_up/down` — диапазон raw payout;
- `payout_correction_up/down` — коррекция конкретного диапазона;
- `features_json.selection_reason` — почему выбрана сторона;
- `features_json.crowd_binance_consensus` — общий сигнал Crowd+Binance либо `null`;
- `features_json.selected_payout_bucket_ready` — готовность выбранного диапазона.

## Обновление проекта

1. Полностью замените все файлы в GitHub.
2. PostgreSQL не удаляйте.
3. Полностью замените Railway Variables новым файлом.
4. Выполните Redeploy.
5. Оставьте одну Railway replica.
6. Проверьте `/health`, `/signal`, `/payout/calibration` и `/strategy/performance`.

## Режим

Только PAPER. Приватный ключ не используется. Реальные транзакции не отправляются.
