# M9 FUSION EV 1.1.0 — PancakeSwap Prediction paper bot

Полный paper-бот для PancakeSwap Prediction. Он объединяет Chainlink/PancakeSwap, Binance, пулы, Bayesian M9 и устойчивые паттерны, но принимает решение по **скорректированному EV**, а не по одному направлению или текущему коэффициенту.

## Главное изменение 1.1.0

Первая версия завышала коэффициент в T−40, потому что пулы продолжали сильно меняться до LOCK. В 1.1.0 добавлены:

- начальная коррекция прогнозируемого коэффициента `0.75`;
- автоматическое обучение коррекции отдельно для UP и DOWN;
- обучение по фактическим финальным пулам PancakeSwap;
- нижний квартиль истории вместо среднего, чтобы не переоценивать payout;
- при отрицательном EV выбор возвращается к более вероятной стороне;
- при слабом согласии моделей payout-кандидат используется только при достаточном скорректированном EV;
- ставки `$10/$15` запрещены, пока нет минимум 200 сделок и готовой статистики payout;
- банк в момент решения и банк перед settlement хранятся отдельно;
- сохраняются фактические коэффициенты обеих сторон и реальный EV выбранного сигнала.

История существующей PostgreSQL **не очищается**. При первом запуске 1.1.0 база автоматически обновит структуру и рассчитает фактические коэффициенты для старых закрытых сделок.

## Логика источников

- Chainlink + движение live-раунда PancakeSwap — базовый вес 40%;
- Binance microstructure — 15%;
- динамика пулов PancakeSwap — 20%;
- Bayesian M9 — 20%, но реальный вес уменьшается при малом числе совпадений;
- устойчивые паттерны — 5%.

Вес каждого доступного источника дополнительно корректируется по Brier score.

## Выбор стороны

1. Рассчитывается вероятность UP/DOWN.
2. Из текущих пулов строится `raw_expected_coeff`.
3. К нему применяется адаптивная коррекция payout.
4. Рассчитывается EV обеих сторон.
5. Если лучший EV отрицательный — выбирается более вероятная сторона и ставка `$5`.
6. Если EV предлагает идти против вероятности, но согласие этой стороны ниже 20% и EV ниже 5% — выбирается более вероятная сторона.
7. Ставка создаётся каждый раунд.

## Ставки

- до 200 закрытых сделок: только `$5`;
- пока payout-калибровка не получила минимум 50 примеров на UP и DOWN: только `$5`;
- после готовности: `$10` при EV ≥ 5% и agreement ≥ 60%;
- `$15` при EV ≥ 12% и agreement ≥ 70%;
- Fibonacci отсутствует;
- увеличения после проигрыша нет;
- автоматической остановки по банку нет.

## Замена текущего проекта

Замените **все файлы** в GitHub содержимым этого пакета. PostgreSQL удалять и создавать заново не нужно. После нового deploy приложение выполнит безопасную миграцию `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`.

Используйте одну Railway replica.

## Проверка

```text
/health
/signal
/status?limit=30
/status?history=all&limit=100000
/history/export.csv
/model/performance
/payout/calibration
```

`/payout/calibration` показывает текущую коррекцию, количество примеров и готовность к переменным ставкам.

## Новые поля истории

- `raw_expected_coeff_up/down` — оценка до адаптивной коррекции;
- `payout_correction_up/down` — применённый коэффициент коррекции;
- `expected_coeff_up/down` — итоговый коэффициент для EV;
- `bank_before` — банк в момент создания решения;
- `bank_before_settlement` — фактический банк непосредственно перед закрытием;
- `final_coeff_up/down` — итоговый гипотетический payout обеих сторон;
- `actual_ev_signal` — EV выбранной стороны по фактическому финальному коэффициенту;
- `payout_ratio_signal` — отношение фактического коэффициента к raw-прогнозу.

## История

PostgreSQL хранит решения без автоматического удаления.

```text
/status?history=all&limit=100000
/history/export.csv
```

Если записей больше 100000:

```text
/status?history=all&limit=100000&offset=100000
```
